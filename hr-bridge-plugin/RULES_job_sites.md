# Правило работы с рабочими сайтами (HH.ru / Rabota.by)

_Владелец: Yuliya. Действует постоянно. Обновлено: 2026-07-01._

Стандартная инструкция для Claude по работе с откликами кандидатов через коннектор
`hr-bridge`. Цель — не возвращаться к этим вопросам каждый раз.

## 1. Смена статуса отклика возможна (в т.ч. «Отказ»)

Бэкенд `maverickframe-hh-bridge.onrender.com` умеет менять статус:

```
POST /{provider}/negotiations/{nid}/change_state
body: {"action": "<action>", "message": "<опц. текст кандидату>"}
```

Внутри вызывает HH/Rabota.by API `PUT /negotiations/{action}/{nid}`.

**Допустимые action:** `discard_by_employer` (Отказ), `consider`, `phone_interview`,
`interview`, `assessment`, `offer`, `hired`, `discard_no_interaction`, `discard_vacancy_closed`.
`provider` = `rabota` (по умолчанию) или `hh`.

## 2. Стоящее согласие Yuliya

- Механизм согласован. Не объяснять каждый раз, что смена статуса возможна, и не
  переспрашивать про подход — просто делать.
- Смену статуса и отправку сообщений выполнять только по явному запросу.

## 3. Защита от ошибок (необратимость)

Статусы через API необратимы из интеграции (откат — вручную в интерфейсе сайта).

- Единичное действие (1 кандидат) по явному запросу — сразу.
- Массовое действие: сначала показать список целей + текст (dry-run), применить после «ок».
  Одно подтверждение на пачку. Проверять правильность вакансии.

## 4. Инструменты

- Постоянный MCP-tool `change_status(nid, action, message, provider)` (см. `servers/server.py`).
- Скрипт `hr_status.py` (dry-run по умолчанию, реальное применение с `--confirm`).
- Либо прямой `POST` из п.1.

## 5. Прочее

- Никогда не выводить токены (OAuth, bot_token).
- `nid` и `resume_id` — из `list_responses`.
- Аккаунты: Rabota.by — vacancy 134229811, employer 772836; HH.ru — employer 12669364.
